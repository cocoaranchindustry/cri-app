01 Prérequis et choix du plan Hostinger

Hostinger propose deux familles d'hébergement. Seul le VPS convient pour une application Next.js : l'hébergement mutualisé ("Business Web Hosting") ne fournit pas de runtime Node.js, ni d'accès shell, ni de reverse proxy personnalisé.

Plan recommandé

PlanRAMvCPUStockagePrix (indicatif)Adapté CRI
VPS KVM 12 Go150 Go SSD~6 €/mois✅ (staging)
VPS KVM 24 Go2100 Go SSD~12 €/mois✅ (production)
VPS KVM 48 Go4200 Go SSD~24 €/mois✅ (gros trafic)
Business Web Hosting——200 Go~3 €/mois❌ Next.js impossible
Recommandation : pour le site institutionnel CRI (≈ 5 000 visites/mois attendues), un KVM 2 (4 Go RAM) est confortable. Le KVM 1 suffit pour staging ou basse audience.

Avant de commencer, prépare

  1. Un compte Hostinger avec un VPS actif (commande sur hostinger.fr).
  2. L'adresse IP du VPS (visible dans le panneau hPanel → VPS → Vue d'ensemble).
  3. Un nom de domaine (ex. cri.africa) — acheter chez Hostinger (transfert de zone simplifié) ou chez un registrar externe (Namecheap, OVH, Gandi).
  4. Un accès au repo Git (GitHub, GitLab, Bitbucket) contenant cri-app.
  5. Les credentials Firebase (voir section 10) et la clé API Brevo pour la newsletter.
  6. Un client SSH sur ta machine locale :
    • Windows : PuTTY ou le Terminal moderne (PowerShell 7+ avec ssh natif).
    • macOS / Linux : Terminal intégré.

02 Premier accès SSH et durcissement du serveur

À la livraison, Hostinger fournit un accès root via SSH. On va l'utiliser, puis créer un utilisateur dédié avec sudo et désactiver la connexion directe en root.

Connexion initiale

# Depuis ta machine locale (Windows PowerShell, macOS ou Linux) :
ssh root@<IP_DU_VPS>

# Mot de passe : celui reçu par email de Hostinger
# Accepte la fingerprint du serveur à la première connexion

Créer un utilisateur non-root

# Une fois connecté en root :
adduser cri
# Définir un mot de passe fort (≥ 16 caractères)

# Ajouter l'utilisateur au groupe sudo :
usermod -aG sudo cri

# Vérifier :
groups cri
# → cri : cri sudo

Désactiver la connexion SSH en root

# Toujours en root :
nano /etc/ssh/sshd_config

# Modifier la ligne :
PermitRootLogin yes  →  PermitRootLogin no

# Sauvegarder (Ctrl+O, Entrée, Ctrl+X), puis redémarrer SSH :
systemctl restart sshd

Configurer le pare-feu UFW

ufw default deny incoming
ufw default allow outgoing
ufw allow OpenSSH          # Port 22 — accès SSH
ufw allow "Nginx Full"   # Ports 80 (HTTP) + 443 (HTTPS)
ufw enable
ufw status verbose
⚠ Important : garde ta session SSH actuelle ouverte tant que tu n'as pas testé la reconnexion avec l'utilisateur cri dans un second terminal. Si tu casses la config, la première session te sauvera.

Reconnexion avec le nouvel utilisateur

# Nouvelle fenêtre terminal :
ssh cri@<IP_DU_VPS>

# Vérifier que sudo fonctionne :
sudo whoami
# → root

03 Installer Node.js 20, npm 10, Git, outils système

Le projet exige Node.js ≥ 20 (cf. apps/web/package.json → "engines.node": ">=20.0.0") et npm ≥ 10. On installe la LTS via NodeSource, qui fournit les versions les plus récentes pour Ubuntu.

Paquets système de base

sudo apt update && sudo apt upgrade -y
sudo apt install -y curl git build-essential ufw software-properties-common ca-certificates gnupg lsb-release

# Vérifier la version Ubuntu :
lsb_release -a
# → Ubuntu 22.04 LTS (recommandé)

Node.js 20 LTS via NodeSource

curl -fsSL https://deb.nodesource.com/setup_20.x | sudo -E bash -
sudo apt install -y nodejs

# Vérification :
node --version   # → v20.x.x
npm --version    # → 10.x.x

Outils de déploiement (recommandés)

# PM2 : gestionnaire de process Node.js (daemon + monitoring)
sudo npm install -g pm2

# Uptime monitoring via pm2 (optionnel mais pratique)
pm2 install pm2-logrotate
pm2 set pm2-logrotate:max_size 10M
pm2 set pm2-logrotate:retain 14

# Nginx (sera installé section 8)
# Certbot (sera installé section 9)

04 Cloner le dépôt cri-app et préparer l'environnement

Créer un répertoire de déploiement

sudo mkdir -p /var/www/cri-app
sudo chown cri:cri /var/www/cri-app
cd /var/www/cri-app

Cloner le repo

# Repo public (HTTPS) :
git clone https://github.com/<ton-org>/cri-app.git .

# Repo privé (SSH, ajouter la clé au préalable sur GitHub) :
# ssh-keygen -t ed25519 -C "cri-vps" -f ~/.ssh/id_ed25519
# cat ~/.ssh/id_ed25519.pub  →  ajouter sur GitHub Settings → SSH keys
git clone git@github.com:<ton-org>/cri-app.git .

# Vérifier l'arborescence :
ls -la
# → apps/  functions/  package.json  nixpacks.toml  railway.json  README.md

Installer les dépendances

# npm workspaces : installe TOUT le monorepo d'un coup
npm ci --include=dev --workspaces --include-workspace-root

# Cette étape peut prendre 2-4 minutes (téléchargement + build des natives)
# Si erreur EACCES sur .npm :
#   sudo chown -R $USER:$USER /var/www/cri-app

Créer un swap (si VPS KVM 1 / 2 Go RAM)

# Recommandé pour éviter les OOM pendant npm run build
sudo fallocate -l 2G /swapfile
sudo chmod 600 /swapfile
sudo mkswap /swapfile
sudo swapon /swapfile
echo '/swapfile none swap sw 0 0' | sudo tee -a /etc/fstab

# Vérifier :
free -h
# → Swap: 2.0Gi

05 Configurer les variables d'environnement (.env.local)

Next.js charge automatiquement .env.local au démarrage. Ce fichier ne doit jamais être commité — il contient les secrets de production.

Créer le fichier

cd /var/www/cri-app/apps/web
cp .env.example .env.local
nano .env.local

Variables à renseigner

VariableValeur productionSource
NEXT_PUBLIC_APP_URL https://cocoaranch.com Ton domaine
NEXT_PUBLIC_APP_ENV production —
NEXT_PUBLIC_FIREBASE_* 7 variables Firebase Console → Paramètres → SDK setup
FIREBASE_PRIVATE_KEY Clé complète du service account Firebase Console → Comptes de service → Générer nouvelle clé privée
SESSION_COOKIE_SECRET ≥ 32 caractères aléatoires openssl rand -base64 48
ENCRYPTION_KEY 32 bytes base64 openssl rand -base64 32
BREVO_API_KEY Clé API v3 app.brevo.com → Settings → API Keys
NEXT_PUBLIC_PLAUSIBLE_DOMAIN cocoaranch.com Si Plausible activé
NEXT_PUBLIC_SENTRY_DSN DSN du projet sentry.io → Settings → Client Keys

Génération des secrets

# SESSION_COOKIE_SECRET :
openssl rand -base64 48

# ENCRYPTION_KEY (32 bytes base64) :
openssl rand -base64 32

# FIREBASE_PRIVATE_KEY : copier depuis le fichier JSON téléchargé
# dans Firebase Console, l'ouvrir avec :
cat ~/Downloads/cri-app-firebase-adminsdk-xxxxx.json

Format de FIREBASE_PRIVATE_KEY (critique)

⚠ Piège classique : la clé privée doit conserver ses retours à la ligne. Dans .env.local, entourer la valeur de double quotes et garder les \n littéraux : 
FIREBASE_PRIVATE_KEY="-----BEGIN PRIVATE KEY-----\nMIIE...\n-----END PRIVATE KEY-----\n"

Protéger le fichier

chmod 600 .env.local
ls -la .env.local
# → -rw------- 1 cri cri  1234 ... .env.local

 • Optimiser les images, générer les routes statiques (22 routes)

Test rapide du build

cd apps/web
npm run start

# Le serveur démarre sur le port 3000.
# Depuis un autre terminal :
curl -I http://localhost:3000

07 Installer PM2 et lancer l'application en daemon

PM2 (Process Manager 2) maintient Next.js en vie, le redémarre après crash, log les sorties stdout/stderr, et s'intègre avec systemd pour survivre aux reboots du VPS.

Créer le fichier de configuration PM2

cd /var/www/cri-app
nano ecosystem.config.js

Contenu du fichier :

module.exports = {
  apps: [
    {
      name: "cri-app",
      script: "npm",
      args: "run start --workspace=apps/web",
      cwd: "/var/www/cri-app",
      instances: 1,
      autorestart: true,
      watch: false,
      max_memory_restart: "512M",
      env: {
        NODE_ENV: "production",
        PORT: 3000,
        HOSTNAME: "127.0.0.1"
      },
      error_file: "/var/log/cri-app/error.log",
      out_file: "/var/log/cri-app/out.log",
      merge_logs: true,
      time: true
    }
  ]
};

Créer le répertoire de logs

sudo mkdir -p /var/log/cri-app
sudo chown cri:cri /var/log/cri-app

Démarrer l'application

cd /var/www/cri-app
pm2 start ecosystem.config.js

# Vérifier le statut :
pm2 status
# ┌────┬──────────────────┬─────────┬──────┬────────┬─────────┐
# │ id │ name             │ mode    │ ↺    │ status │ cpu │ mem │
# ├────┼──────────────────┼─────────┼──────┼────────┼─────────┤
# │ 0  │ cri-app          │ fork    │ 0    │ online │ 0%  │ 80M │
# └────┴──────────────────┴─────────┴──────┴────────┴─────────┘

Activer le démarrage automatique (systemd)

pm2 startup systemd
# → Une commande sudo est affichée, copier-coller :

sudo env PATH=$PATH:/usr/bin pm2 startup systemd -u cri --hp /home/cri

pm2 save
# → sauvegarde la liste des process à redémarrer

Commandes utiles PM2

pm2 logs cri-app           # Logs en temps réel (Ctrl+C pour quitter)
pm2 logs cri-app --lines 200  # 200 dernières lignes
pm2 restart cri-app        # Redémarrer après un build
pm2 reload cri-app         # Zero-downtime reload (instances ≥ 2)
pm2 stop cri-app           # Arrêter
pm2 delete cri-app         # Supprimer de PM2
pm2 monit                  # Dashboard CPU/RAM en terminal

08 Installer et configurer Nginx comme reverse proxy

Nginx reçoit les requêtes HTTPS sur le port 443, les termine (SSL), puis les forwarde à Next.js sur le port 3000. Il gère aussi les fichiers statiques, la compression gzip et les headers de sécurité.

Installer Nginx

sudo apt install -y nginx
sudo systemctl enable nginx
sudo systemctl start nginx

# Vérifier que Nginx tourne :
sudo systemctl status nginx
curl -I http://localhost
# → HTTP/1.1 200 OK (page d'accueil Nginx par défaut)

Créer le fichier de configuration du site

sudo nano /etc/nginx/sites-available/cri-app

Contenu (adapté pour le domaine cocoaranch.com) :

# /etc/nginx/sites-available/cri-app
# Reverse proxy pour Cocoa Ranch & Industry (cocoaranch.com)

# Redirection HTTP → HTTPS (gérée par Certbot plus tard)
server {
    listen 80;
    listen [::]:80;
    server_name cocoaranch.com www.cocoaranch.com;

    # Permet à Certbot de valider le challenge ACME
    location /.well-known/acme-challenge/ {
        root /var/www/html;
    }

    location / {
        return 301 https://$host$request_uri;
    }
}

# Serveur HTTPS principal
server {
    listen 443 ssl http2;
    listen [::]:443 ssl http2;
    server_name cocoaranch.com www.cocoaranch.com;

    # Logs
    access_log /var/log/nginx/cri-app.access.log;
    error_log  /var/log/nginx/cri-app.error.log;

    # Limite de taille d'upload (formulaires de contact, etc.)
    client_max_body_size 10M;

    # Proxy vers Next.js
    location / {
        proxy_pass http://127.0.0.1:3000;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection 'upgrade';
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_cache_bypass $http_upgrade;
        proxy_read_timeout 60s;
        proxy_send_timeout 60s;
    }

    # Cache agressif pour les assets statiques de Next.js
    location /_next/static/ {
        proxy_pass http://127.0.0.1:3000;
        proxy_cache_valid 200 365d;
        add_header Cache-Control "public, max-age=31536000, immutable";
        access_log off;
    }

    # Compression gzip
    gzip on;
    gzip_vary on;
    gzip_min_length 1024;
    gzip_types text/plain text/css text/xml text/javascript
               application/json application/javascript application/xml+rss
               application/atom+xml image/svg+xml;
}

Activer le site

# Créer le symlink (Nginx lit uniquement sites-enabled) :
sudo ln -s /etc/nginx/sites-available/cri-app /etc/nginx/sites-enabled/

# Supprimer le site par défaut :
sudo rm /etc/nginx/sites-enabled/default

# Tester la syntaxe :
sudo nginx -t
# → syntax is ok / test is successful

# Recharger :
sudo systemctl reload nginx

Tester sans HTTPS (HTTP seulement pour l'instant)

# Depuis ton PC local :
curl -I http://<IP_DU_VPS>
# → HTTP/1.1 301 Moved Permanently  (redirection vers HTTPS)

# Une vraie URL type :
curl -I -L http://<IP_DU_VPS>/projet
# Le -L suit la redirection, et tu obtiens 200 si Next.js répond bien

09 Activer HTTPS avec Let's Encrypt (Certbot)

Let's Encrypt fournit des certificats SSL gratuits et renouvelables automatiquement. Certbot est l'outil officiel fourni par l'EFF (Electronic Frontier Foundation).

Prérequis DNS : avant de demander un certificat, le domaine cocoaranch.com doit pointer vers l'IP du VPS. Voir section 11. Si tu n'as pas encore configuré les DNS, saute à cette section puis reviens ici.

Installer Certbot

sudo apt install -y certbot python3-certbot-nginx

Demander un certificat

sudo certbot --nginx -d cocoaranch.com -d www.cocoaranch.com

# Certbot va demander :
# 1. Une adresse email (pour les notifications d'expiration)
# 2. Accepter les conditions de Let's Encrypt (A)gree
# 3. Partager ou non ton email avec l'EFF (Y/N)
# 4. Rediriger tout le trafic HTTP vers HTTPS ?  →  2 (oui, automatique)

# En sortie :
# Successfully received certificate.
# Certificate is saved at: /etc/letsencrypt/live/cocoaranch.com/fullchain.pem
# Key is saved at:         /etc/letsencrypt/live/cocoaranch.com/privkey.pem
# These files will be updated when the certificate renews.

Vérifier le renouvellement automatique

# Test à blanc :
sudo certbot renew --dry-run
# → Congratulations, all simulated renewals succeeded

# Un timer systemd est installé automatiquement par le package
# (certbot.timer) pour renouveler le cert 30 jours avant expiration.
systemctl list-timers | grep certbot

Headers de sécurité ajoutés par Certbot

Certbot a modifié /etc/nginx/sites-available/cri-app pour ajouter automatiquement :

  • ssl_certificate + ssl_certificate_key pointant vers Let's Encrypt
  • ssl_protocols TLSv1.2 TLSv1.3
  • Header Strict-Transport-Security (HSTS)

Durcir la configuration SSL (recommandé)

sudo nano /etc/nginx/snippets/ssl-params.conf

# Contenu :
ssl_protocols TLSv1.2 TLSv1.3;
ssl_prefer_server_ciphers on;
ssl_ciphers ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256:ECDHE-ECDSA-AES256-GCM-SHA384:ECDHE-RSA-AES256-GCM-SHA384;
ssl_session_cache shared:SSL:10m;
ssl_session_timeout 1d;
ssl_session_tickets off;
ssl_stapling on;
ssl_stapling_verify on;
add_header Strict-Transport-Security "max-age=63072000; includeSubDomains; preload" always;
add_header X-Frame-Options "DENY" always;
add_header X-Content-Type-Options "nosniff" always;
add_header Referrer-Policy "strict-origin-when-cross-origin" always;

Inclure ce fichier dans le bloc server { ... } HTTPS :

server {
    listen 443 ssl http2;
    server_name cocoaranch.com;
    include /etc/nginx/snippets/ssl-params.conf;  # ← ajouter cette ligne
    ssl_certificate /etc/letsencrypt/live/cocoaranch.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/cocoaranch.com/privkey.pem;
    # ...
}

sudo nginx -t && sudo systemctl reload nginx

Test final

# Depuis ton PC :
curl -I https://cocoaranch.com
# → HTTP/2 200

# Vérifier la note SSL :
# → https://www.ssllabs.com/ssltest/analyze.html?d=cocoaranch.com
# Cible : note A ou A+

10 Configurer Firebase Authentication (domaines autorisés)

Firebase Auth refuse par défaut les connexions depuis des domaines non whitelistés. Il faut autoriser cocoaranch.com et www.cocoaranch.com.

Dans Firebase Console

  1. Aller sur console.firebase.google.com et ouvrir le projet CRI.
  2. Authentication → Settings → Authorized domains
  3. Cliquer "Add domain" et ajouter :
    • cocoaranch.com
    • www.cocoaranch.com
    • (optionnel) le sous-domaine de staging : staging.cocoaranch.com
  4. Vérifier le projet Firebase → Paramètres → Général → "Your project" → Project ID = cri-app (doit correspondre à FIREBASE_PROJECT_ID dans .env.local).
  5. Récupérer les clés Admin SDK (pour Firebase Admin côté serveur) :
    • Paramètres du projet → Comptes de service → Générer une nouvelle clé privée
    • Le JSON téléchargé contient : project_id, client_email, private_key
    • Copier dans .env.local (cf. section 5)

Vérifier les Firestore Rules

Les règles Firestore sont déployées séparément :

# Depuis ta machine locale :
cd cri-app
firebase deploy --only firestore:rules,firestore:indexes,storage

# Les fichiers à commiter sont à la racine du monorepo :
#   firestore.rules  (RBAC + validation multi-rôles)
#   firestore.indexes.json
#   storage.rules  (validation MIME + taille)

11 Pointer le domaine cocoaranch.com vers le VPS

Ton domaine cocoaranch.com doit résoudre vers l'IP du VPS. Selon où tu as acheté le domaine, la procédure diffère légèrement.

Récupérer l'IP du VPS

Dans le panneau hPanel de Hostinger : VPS → Vue d'ensemble → "Adresse IP". Exemple : 195.201.42.17

Si le domaine est chez Hostinger (hPanel)

  1. hPanel → Domaines → cocoaranch.com → DNS / Zone DNS
  2. Ajouter / modifier l'enregistrement A :
    • Type : A
    • Nom : @ (racine)
    • Valeur : 195.201.42.17 (ton IP)
    • TTL : 14400 (4h, raisonnable)
  3. Ajouter un enregistrement CNAME pour www :
    • Type : CNAME
    • Nom : www
    • Valeur : cocoaranch.com

Si le domaine est chez un registrar externe (OVH, Namecheap, Gandi, GoDaddy)

  1. Connecte-toi à ton registrar
  2. Ouvre la zone DNS de cocoaranch.com
  3. Même configuration que ci-dessus (enregistrement A + CNAME)

Vérifier la propagation DNS

# Outils en ligne :
# → https://dnschecker.org (vérifie la propagation mondiale)
# → https://www.whatsmydns.net

# En local (Linux/macOS) :
dig cocoaranch.com A +short
# → 195.201.42.17

nslookup cocoaranch.com

# Délai de propagation : 1 à 48h selon les TLD et le registrar.
# .com se propage généralement en 1-4h.
Pour le sous-domaine www : ne crée pas un second enregistrement A, utilise un CNAME vers cocoaranch.com. C'est plus propre et plus simple à maintenir.

12 Sauvegardes, mises à jour, monitoring

Mettre à jour le code (workflow de déploiement)

cd /var/www/cri-app

# 1. Récupérer les derniers commits :
git pull origin main

# 2. Installer les nouvelles dépendances (si package.json a changé) :

# 3. Rebuild :
npm run build

# 4. Relancer l'app (zero-downtime) :

# 5. Vérifier :

curl -I https://cri.africa

Pour automatiser, tu peux créer un script /var/www/cri-app/deploy.sh :

#!/bin/bash
set -e
cd /var/www/cri-app
echo "[$(date)] Pulling latest code..."
git pull origin main
echo "[$(date)] Installing deps..."
npm ci --include=dev --workspaces --include-workspace-root
echo "[$(date)] Building..."
npm run build
echo "[$(date)] Reloading PM2..."
pm2 reload cri-app
echo "[$(date)] Deploy OK"
chmod +x deploy.sh
./deploy.sh

Mises à jour système

# Hebdomadaire (cron automatique possible) :
sudo apt update && sudo apt upgrade -y
sudo reboot   # PM2 relancera Next.js automatiquement après le reboot

Sauvegardes

scp /home/cri/backup-*.tar.gz cri@backup.cri.africa:~/backups/

  • Better Stack (10 moniteurs gratuits, joli dashboard)

Configurer une alerte email + SMS si https://cocoaranch.com ne répond plus pendant > 5 min.

13 Dépannage des erreurs courantes

1. "502 Bad Gateway" après le déploiement

Next.js n'est pas lancé, ou a planté.

pm2 status            # Vérifier le statut
pm2 logs cri-app      # Lire les logs
pm2 restart cri-app   # Redémarrer

# Si PM2 ne le voit plus du tout :
cd /var/www/cri-app
pm2 start ecosystem.config.js

2. Erreurs CORS sur Firebase / Firestore

Le domaine n'est pas autorisé dans Firebase Console (cf. section 10). Vérifier l'orthographe exacte et attendre 5 min après l'ajout (propagation Firebase).

3. "Cannot find module './vendor-chunks/...'"

Bug connu Windows + Next.js 14.2. Cache .next/ corrompu.

cd /var/www/cri-app
rm -rf apps/web/.next
npm run build

4. OOM (Out of Memory) pendant le build

Le VPS manque de RAM. Le swap aide, sinon il faut upgrader.

# Vérifier la RAM dispo :
free -h
dmesg | grep -i "killed process"  # confirmer un OOM

# Augmenter le swap (de 2 à 4 Go) :
sudo swapoff /swapfile
sudo fallocate -l 4G /swapfile
sudo chmod 600 /swapfile
sudo mkswap /swapfile
sudo swapon /swapfile

5. Certificat SSL expiré

Le timer systemd n'a pas tourné. Forcer le renouvellement :

sudo certbot renew --force-renewal
sudo systemctl reload nginx

6. Le site est lent / time out

Vérifier les ressources utilisées :

pm2 monit                  # CPU/RAM Next.js
htop                       # Processus système (sudo apt install htop)
sudo tail -f /var/log/nginx/cri-app.error.log  # Erreurs Nginx
sudo nginx -T 2> /dev/null | grep -i timeout  # Timeouts

7. Mise à jour Node.js requise

Si npm refuse avec un warning de version, mettre à jour :

curl -fsSL https://deb.nodesource.com/setup_22.x | sudo -E bash -
sudo apt install -y nodejs
node --version  # → v22.x

# Reconstruire le projet :
cd /var/www/cri-app
rm -rf node_modules apps/*/node_modules package-lock.json apps/*/package-lock.json
npm install
npm run build
pm2 reload cri-app
