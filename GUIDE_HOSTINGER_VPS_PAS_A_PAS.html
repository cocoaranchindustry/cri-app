01 Comprendre ce qu'on va faire

Avant de taper la moindre commande, prends 5 minutes pour comprendre pourquoi on fait tout ça.

L'analogie de la boutique

Imagine que tu ouvres une boutique de cacao à Douala :

  • Le VPS, c'est le local commercial (un espace vide que tu loues, où personne ne passe encore).
  • Node.js + PM2, c'est ton vendeur qui connaît les produits et sert les clients.
  • Nginx, c'est l'hôte d'accueil à la porte : il redirige les gens vers le bon vendeur, gère la file d'attente, et fait respecter les règles de sécurité.
  • Let's Encrypt (SSL), c'est le cadenas sur la porte et la vitrine protégée — les clients voient que c'est un commerce sérieux (cadenas vert dans le navigateur).
  • Le DNS, c'est l'adresse postale + le panneau dans la rue : sans ça, personne ne sait où te trouver.

Le parcours d'un visiteur

Quand quelqu'un tape https://cocoaranchindustry.cloud dans son navigateur :

  1. Son ordinateur demande : "c'est quoi l'IP de cocoaranchindustry.cloud ?" → DNS répond avec l'IP du VPS.
  2. Le navigateur se connecte en HTTPS à cette IP, sur le port 443.
  3. Le port 443 arrive sur Nginx (le standardiste).
  4. Nginx vérifie le certificat SSL (le cadenas).
  5. Nginx transmet la requête à Next.js (port 3000), lancé par PM2.
  6. Next.js génère la page HTML, la renvoie à Nginx, qui la renvoie au visiteur.
💡 Ce qu'il faut retenir
Tu n'as rien à coder : c'est de la configuration système. Chaque commande fait UNE chose simple. L'astuce, c'est de les taper dans le bon ordre.

06 Sécuriser le serveur

Par défaut, tu es connecté en root (l'administrateur suprême). C'est dangereux car une seule erreur peut tout casser. On va créer un utilisateur dédié et couper l'accès root.

6.1 — Créer un utilisateur "cri"

adduser cri

Le système te demande un mot de passe — choisis un truc solide (16+ caractères). Pour les autres questions (nom, téléphone...), appuie sur Entrée pour laisser vide.

6.2 — Lui donner les droits administrateur

usermod -aG sudo cri

Ainsi cri peut faire des commandes admin avec le préfixe sudo.

6.3 — Couper l'accès SSH en root

nano /etc/ssh/sshd_config

Dans le fichier, cherche la ligne PermitRootLogin yes et change-la en PermitRootLogin no. Pour naviguer dans nano : Ctrl+W pour chercher, flèches pour bouger.

Une fois modifié : Ctrl+O (sauver), Entrée, puis Ctrl+X (quitter).

Applique la modification :

systemctl restart sshd

6.4 — Configurer le pare-feu

Le pare-feu bloque tout le trafic entrant sauf ce qu'on autorise. C'est comme un vigile à l'entrée du bâtiment.

ufw default deny incoming
ufw default allow outgoing
ufw allow OpenSSH
ufw allow "Nginx Full"
ufw enable
ufw status

À la question "Command may disrupt existing ssh connections", tape y.

6.5 — Mettre à jour le système

apt update && apt upgrade -y

Ça télécharge les derniers correctifs de sécurité. Ça peut prendre 1-3 minutes.

6.6 — Reconnexion en tant que "cri"

Important : ne ferme pas ta session root actuelle tant que tu n'as pas testé la nouvelle.

Ouvre un deuxième terminal et tape :

ssh cri@187.55.229.153

Si ça marche → tu es connecté en cri@vps:~$. Tu peux maintenant fermer le premier terminal root.

🚨 À partir de maintenant
Toutes les commandes admin sont précédées de sudo. Par exemple sudo apt update au lieu de apt update.

9.4 — Le piège du FIREBASE_PRIVATE_KEY

La clé privée Firebase est la seule variable qui pose souvent problème. Elle doit être entourée de guillemets et contenir des \n littéraux (barre oblique + n) :

FIREBASE_PRIVATE_KEY="-----BEGIN PRIVATE KEY-----\nMIIEvQIBADANBgkqh...\n-----END PRIVATE KEY-----\n"
🚨 3 pièges à éviter
  1. Ne pas mettre de vrais retours à la ligne dans la valeur (juste \n).
  2. Bien mettre les guillemets au début et à la fin.
  3. Ne pas avoir d'espace avant/après le = dans certains cas (suivre le format KEY=value).

9.5 — Sauvegarder et verrouiller

Dans nano : Ctrl+O, Entrée, puis Ctrl+X.

Ensuite :

chmod 600 .env.local
ls -la .env.local

Tu dois voir -rw------- au début (seul toi peux lire ce fichier).

10.5 — Si erreur OOM : créer un swap

Un swap, c'est un fichier sur le disque que Linux utilise comme "RAM de secours" quand la vraie RAM est pleine.

sudo fallocate -l 2G /swapfile
sudo chmod 600 /swapfile
sudo mkswap /swapfile
sudo swapon /swapfile
echo '/swapfile none swap sw 0 0' | sudo tee -a /etc/fstab
free -h

La dernière commande doit montrer Swap: 2.0Gi.

12 Installer Nginx (le standardiste)

Nginx est un "reverse proxy" : il reçoit les visiteurs sur le port 80/443 et les passe à Next.js (port 3000). Il gère aussi la sécurité, le cache, et la compression.

12.1 — Installer Nginx

sudo apt install -y nginx
sudo systemctl enable nginx
sudo systemctl start nginx

Vérifie :

curl -I http://187.55.229.153

Tu dois voir la page d'accueil Nginx par défaut (HTTP 200).

12.2 — Créer la config pour cocoaranchindustry.cloud

sudo nano /etc/nginx/sites-available/cri-app

Copie-colle ce contenu :

server {
    listen 80;
    server_name cocoaranchindustry.cloud www.cocoaranchindustry.cloud;

    location / {
        return 301 https://$host$request_uri;
    }
}

server {
    listen 443 ssl http2;
    server_name cocoaranchindustry.cloud www.cocoaranchindustry.cloud;

    access_log /var/log/nginx/cri-app.access.log;
    error_log  /var/log/nginx/cri-app.error.log;

    client_max_body_size 10M;

    location / {
        proxy_pass http://127.0.0.1:3000;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection 'upgrade';
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_cache_bypass $http_upgrade;
    }

    location /_next/static/ {
        proxy_pass http://127.0.0.1:3000;
        proxy_cache_valid 200 365d;
        add_header Cache-Control "public, max-age=31536000, immutable";
    }

    gzip on;
    gzip_types text/plain text/css application/json application/javascript text/xml application/xml;
}

Sauvegarde : Ctrl+O, Entrée, Ctrl+X.

12.3 — Activer la config et désactiver le site par défaut

sudo ln -s /etc/nginx/sites-available/cri-app /etc/nginx/sites-enabled/
sudo rm /etc/nginx/sites-enabled/default
sudo nginx -t

Si la dernière commande affiche syntax is ok / test is successful, tout va bien.

13 Activer le HTTPS (cadenas vert 🔒)

Le HTTPS crypte la communication entre le visiteur et le serveur. Sans ça, les navigateurs affichent "Site non sécurisé". Let's Encrypt fournit des certificats gratuits.

13.1 — Installer Certbot

Certbot est l'outil officiel pour obtenir et renouveler les certificats Let's Encrypt.

sudo apt install -y certbot python3-certbot-nginx

13.2 — Demander le certificat

⚠ Prérequis
Le domaine cocoaranchindustry.cloud doit déjà pointer vers l'IP du serveur (étape 14). Si ce n'est pas encore fait, fais-le d'abord. 
sudo certbot --nginx -d cocoaranchindustry.cloud -d www.cocoaranchindustry.cloud

Réponds aux questions :

  1. Email : ton email (pour recevoir les alertes d'expiration).
  2. Accepter les conditions : tape Y.
  3. Partager ton email avec l'EFF : tape N (ou Y, pas critique).
  4. Rediriger HTTP vers HTTPS ? : tape 2 (oui, automatique).

13.3 — Vérifier le résultat

Si tout va bien, Certbot affiche :

Congratulations! You have successfully enabled HTTPS on
https://cocoaranchindustry.cloud and https://www.cocoaranchindustry.cloud

13.4 — Tester depuis ton PC

Ouvre un navigateur et va sur https://cocoaranchindustry.cloud. Tu dois voir le cadenas vert 🔒 et le site s'afficher.

13.5 — Le renouvellement automatique

Les certificats Let's Encrypt expirent tous les 90 jours, mais un timer systemd les renouvelle automatiquement. Vérifie :

sudo certbot renew --dry-run

Si ça affiche Congratulations, all simulated renewals succeeded, tu n'as rien à faire de plus.

14 Faire pointer cocoaranchindustry.cloud vers le serveur

Le DNS, c'est l'annuaire d'Internet. Quand quelqu'un tape cocoaranchindustry.cloud, son ordinateur demande : "c'est quoi l'IP de ce domaine ?" Si tu ne configures pas le DNS, personne ne trouve ton serveur.

14.1 — Récupérer l'IP du VPS

Tu l'as déjà à l'étape 3 : 187.55.229.153.

14.2 — Configurer le DNS

Ça dépend de l'endroit où tu as acheté le domaine cocoaranchindustry.cloud :

Option A — le domaine est chez Hostinger (hPanel)

  1. Va sur hpanel.hostinger.com.
  2. Menu Domaines → clique sur cocoaranchindustry.cloud.
  3. Onglet DNS / Zone DNS.
  4. Cherche l'enregistrement de type A avec le nom @ (ou ajoute-en un).
    • Type : A

17 Que faire si ça ne marche pas ?

🔴 "502 Bad Gateway"

Next.js ne tourne pas ou a planté.

pm2 status
pm2 logs cri-app --lines 50
pm2 restart cri-app

🔴 Erreur Firebase "auth/unauthorized-domain"

Le domaine n'est pas autorisé. Retourne à l'étape 15.

🔴 Le site est très lent

pm2 monit
htop
df -h /var/www

Si la RAM est pleine, le VPS est sous-dimensionné. Passe au KVM 4 ou KVM 8.

🔴 SSL expiré (alerte par email)

sudo certbot renew --force-renewal
sudo systemctl reload nginx

🔴 Tu as perdu l'accès SSH

Hostinger propose un accès "console" via le navigateur dans hPanel (VPS → Console). Tu peux te connecter même si le réseau bloque le port 22.

🔴 Oubli du mot de passe d'un utilisateur

Via la console Hostinger :

sudo passwd cri

🔴 Tu t'es fait déconnecter et tu n'arrives plus à te reconnecter en SSH

Hostinger propose une console de secours directement dans le navigateur : hPanel → VPS → Console. C'est comme un terminal branché en direct sur la machine, qui marche même si le réseau bloque le port 22.

🔴 Le DNS ne se propage pas (nslookup renvoie une vieille IP)

Vide le cache DNS de ton PC :

# Windows (PowerShell admin) :
ipconfig /flushdns

# macOS :
sudo dscacheutil -flushcache

# Linux (systemd-resolved) :
sudo resolvectl flush-caches

Si après 24h c'est toujours pas propagé, vérifie l'orthographe exacte dans la zone DNS (sans faute, sans "www" parasite).

🔴 Le HTTPS ne marche pas alors que Nginx tourne

Si https://cocoaranchindustry.cloud donne "connexion refusée" :

# Vérifier que le firewall laisse passer le HTTPS :
sudo ufw status
# → doit afficher "Nginx Full" ou "443/tcp ALLOW"

# Vérifier que Nginx écoute bien sur 443 :
sudo ss -tlnp | grep :443
# → doit afficher "nginx" et l'IP 0.0.0.0:443
